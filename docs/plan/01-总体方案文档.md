# mc-netprobe 总体方案文档

## 1. 目标

本项目用于对以下 Minecraft 服务拓扑进行自动化、可重复、可归档的链路质量测试：

`客户端 -> FRP 公网服务器 -> FRP 隧道 -> Mac mini(MC服务器)`

目标不是单点测速，而是同时度量：

1. 总链路质量  
   客户端到 MC 实际公网入口的整体网络表现。

2. 分段链路质量  
   - 分段 A：客户端 -> FRP 公网服务器  
   - 分段 B：FRP 公网服务器 -> Mac mini

3. 本机对照质量  
   Mac mini 本地到本机 MC 服务，用于排除 MC 服务端或主机自身问题。

4. 带载情况下的体验退化  
   例如上/下行被占满时，MC 端口的 TCP 建连延迟、抖动、失败率是否明显恶化。

最终产出应包括：

- 结构化原始结果（JSON）
- 聚合结果（CSV）
- 人可读报告（HTML/Markdown）
- 阈值告警与瓶颈定位结论

---

## 2. 设计原则

### 2.1 单仓库、单目录树

服务端 agent、FRP 公网服务器 agent、客户端 agent、控制器、配置、结果导出统一放在一个仓库内，便于：

- Codex 一次性生成
- 一键安装与启动
- 统一版本管理
- 统一结果格式

### 2.2 以控制器为中心

由控制器统一读取配置、通过 SSH 触发远端 agent、安排测试顺序、收集结果、汇总结论。避免三端手工分别执行脚本。

### 2.3 先做 M VP，再逐步增强

第一版优先实现：

- ping/ICMP 统计
- TCP connect 探测
- iperf3 吞吐探测
- 系统负载采集
- SSH 远程执行
- JSON/CSV/HTML 报告

后续再增强：

- MC status 协议探测
- traceroute/mtr 深度路径观测
- 图表化 HTML 报告
- 定时回归测试

### 2.4 指标以“体验影响”为导向

优先关注会影响游戏手感的指标：

- RTT 均值、P95、P99
- 抖动
- 丢包
- TCP 建连时间与失败率
- 吞吐期间的延迟膨胀
- 带载时的超时和重传倾向
- MC 应用层响应时间

## 3. 目标拓扑与测量点

### 3.1 逻辑拓扑

- 客户端：真实玩家机器或测试机
- relay：FRP 公网服务器
- server：Mac mini，承载 MC 服务器
- mc_public：玩家实际连接的公网入口（域名/IP:25565）
- iperf_public：专门为测速映射的测试端口，例如 5201

### 3.2 链路划分

#### 总链路
- 客户端 -> mc_public

#### 分段 A
- 客户端 -> relay

#### 分段 B
- relay -> server

#### 本地对照
- server -> 127.0.0.1:25565

---

## 4. 测试指标定义

## 4.1 基础网络层

用于度量路径的基础可达性与时延特征。

指标：

- `packet_loss_pct`
- `rtt_min_ms`
- `rtt_avg_ms`
- `rtt_p95_ms`
- `rtt_p99_ms`
- `rtt_max_ms`
- `jitter_ms`

建议实现：

- `ping`
- 后续可选 `mtr`

适用路径：

- client -> relay
- relay -> server
- 可选 client -> mc_public（若公网入口允许 ICMP）

---

## 4.2 TCP 连接层

用于逼近 MC 使用的 TCP 长连接质量。

指标：

- `connect_success_rate_pct`
- `connect_timeout_or_error_pct`
- `connect_avg_ms`
- `connect_p95_ms`
- `connect_p99_ms`
- `connect_max_ms`

建议实现：

- 多次短连接采样
- 并发和串行均可配置
- 间隔默认 200ms 到 500ms

适用路径：

- client -> relay 的测试端口
- relay -> server 的 MC / iperf3 端口
- client -> mc_public:25565

---

## 4.3 吞吐层

用于测量带宽能力及其对体验的影响。

指标：

- `throughput_up_mbps`
- `throughput_down_mbps`
- `throughput_stability_score`
- `retransmits`
- `test_duration_sec`

建议实现：

- 使用 `iperf3`
- forward 和 reverse 都要测
- 默认 30 秒
- 可配置并发流数，第一版先单流

适用路径：

- client -> relay
- relay -> server
- client -> iperf_public（经 FRP 映射）
- 可选 server 本地回环对照

---

## 4.4 带载退化层

这是最关键的指标之一。因为很多家庭网络在空载时表现正常，但一旦上传/下载被占满，延迟就会暴涨。

指标：

- `idle_connect_avg_ms`
- `loaded_connect_avg_ms`
- `load_rtt_inflation_ms`
- `loaded_connect_p95_ms`
- `loaded_timeout_pct`

测试方式：

1. 启动一条 `iperf3` 压测
2. 同时每 500ms 对 `mc_public:25565` 做 TCP connect 采样
3. 比较空载与满载时的差异

适用路径：

- client -> mc_public
- client -> relay
- relay -> server

---

## 4.5 应用层（MC）

第一版最小实现：

- `mc_tcp_connect_avg_ms`
- `mc_tcp_connect_p95_ms`
- `mc_tcp_connect_success_rate_pct`

第二版增强实现：

- MC status ping/query
- `mc_status_avg_ms`
- `mc_status_p95_ms`
- `mc_status_error_pct`

---

## 4.6 系统辅助指标

用于辅助定位瓶颈是否来自主机资源或 FRP 进程本身。

Mac mini / relay 都应采集：

- CPU 使用率
- 内存使用率
- load average
- 网卡上/下行速率
- 磁盘 IO（可选）
- 关键进程 CPU / RSS（FRP、java）

---

## 5. 测试矩阵

### 5.1 客户端侧执行

1. 对 relay 做 ping
2. 对 relay 测试端口做 TCP connect
3. 对 mc_public:25565 做 TCP connect
4. 对 iperf_public:5201 做上下行吞吐
5. 吞吐期间并发测 mc_public:25565 的 TCP connect
6. 可选 MC status 探测

### 5.2 relay 侧执行

1. 对 server 做 ping
2. 对 server:25565 做 TCP connect
3. 对 server:5201 做吞吐
4. 采集 FRP 进程与系统负载

### 5.3 server 侧执行

1. 对本地 127.0.0.1:25565 做 TCP connect
2. 启动 iperf3 server
3. 采集主机和 MC 进程负载

## 7. 配置设计

### 7.1 topology.yaml

应至少定义：

- client / relay / server 三个节点
- mc_public
- iperf_public
- ssh 登录参数
- 端口配置
- 测试计划参数

示例字段：

```yaml
project_name: mc-frp-netprobe

nodes:
  client:
    role: client
    host: 127.0.0.1
  relay:
    role: relay
    host: relay.example.com
    ssh_user: ubuntu
    ssh_port: 22
  server:
    role: server
    host: 192.168.1.20
    ssh_user: myuser
    ssh_port: 22

services:
  mc_public:
    host: play.example.com
    port: 25565
  iperf_public:
    host: play.example.com
    port: 5201
  mc_local:
    host: 127.0.0.1
    port: 25565
  iperf_local:
    host: 0.0.0.0
    port: 5201
```

### 7.2 thresholds.yaml

用于告警与报告高亮。

建议定义：

- RTT 均值上限
- P95 上限
- 抖动上限
- 丢包上限
- TCP 建连均值上限
- 带载膨胀上限
- 吞吐下限
- MC 响应时延上限

## 8. 运行流程

### 8.1 阶段 1：环境检查

控制器启动后先做：

- Python 版本检查
- `iperf3` 检查
- `ping` 检查
- `ssh` 检查
- 远程 agent 目录与虚拟环境检查
- mc_public 和 iperf_public 端口连通性检查

### 8.2 阶段 2：基线采样

执行：

- client -> relay ping
- client -> relay tcp connect
- relay -> server ping
- relay -> server tcp connect
- client -> mc_public tcp connect
- server -> mc_local tcp connect

### 8.3 阶段 3：吞吐测试

执行：

- relay -> server iperf forward/reverse
- client -> iperf_public iperf forward/reverse
- 可选 client -> relay iperf forward/reverse

### 8.4 阶段 4：带载退化测试

执行：

- 先做空载 TCP 基线
- 然后对目标跑 iperf3 压测
- 压测同时对 MC 端口做持续 TCP 采样
- 计算带载膨胀

### 8.5 阶段 5：系统指标采集

执行：

- relay 系统快照
- server 系统快照
- 可选进程快照（frps/frpc/java）

### 8.6 阶段 6：导出结果

产出：

- `raw.json`
- `summary.csv`
- `report.html`

## 9. 控制器职责

控制器的职责包括：

1. 读取配置  
2. 校验配置  
3. 建立 SSH 连接  
4. 远程触发 agent  
5. 安排测试顺序  
6. 收集结果  
7. 聚合与比对阈值  
8. 导出报告

控制器不应承担复杂采样逻辑，复杂逻辑应落在 `probes/` 中。

---

## 10. agent 设计

### 10.1 设计原则

agent 应保持轻量，最好支持“单任务执行模式”，便于通过 SSH 直接触发。

例如：

```bash
python -m agents.agent_server --task tcp_probe --host 127.0.0.1 --port 25565
python -m agents.agent_server --task start_iperf_server --port 5201
python -m agents.agent_relay --task ping --host 192.168.1.20 --count 50
```

### 10.2 不建议第一版引入复杂常驻服务

第一版先用 SSH + 命令式执行即可，部署复杂度最低。

---

## 11. 推荐测试顺序

推荐顺序如下：

1. server 启动 iperf3 server
2. client 测 relay 基础质量
3. relay 测 server 基础质量
4. client 测 mc_public 基础质量
5. relay 测 server 吞吐
6. client 测 iperf_public 吞吐
7. client 做“压测 + MC 端口延迟观测”
8. 采集 server / relay 系统状态
9. 汇总报告

原因：

- 先做基线，后做压测，避免压测污染基线
- 先测分段，再测总链路，便于定位
- 带载退化测试放到后面，防止影响其他结果

## 13. 报告结论结构

HTML/Markdown 报告建议分成三部分：

### 13.1 总链路结论
例如：

- 总链路平均 TCP 建连延迟 34ms，P95 52ms，空载表现正常
- 吞吐压测时 MC 端口延迟膨胀至 126ms，带载风险明显

### 13.2 分段瓶颈定位
例如：

- client -> relay 时延与抖动正常
- relay -> server 抖动与丢包偏高
- 瓶颈主要位于 FRP 公网服务器到家庭网络出口这一段

### 13.3 处置建议
例如：

- 限制上行带宽使用
- 在家庭路由器启用 SQM/QoS
- 升级 FRP 公网服务器线路或机房
- 关闭 FRP 压缩，观察小包延迟
- 检查 Mac mini 侧 Java/磁盘负载

## 14. 风险与边界

### 14.1 ICMP 不能代表全部体验
某些网络会限速 ICMP，因此 ping 正常不等于游戏体验正常。

### 14.2 带宽不是唯一瓶颈
MC 单玩家并不一定需要很高带宽。通常更关键的是抖动、丢包、排队和主机负载。

### 14.3 FRP 端口映射需专门开放测速口
不要只测 25565。必须给 `iperf3` 独立映射端口，例如 5201。

### 14.4 不同时间段表现会明显不同
应把测试时段写入结果，最好在晚高峰与低峰各测一次。

## 17. 验收标准

完成后至少满足：

1. 单命令启动完整测试  
2. 能同时产出总链路和分段链路结果  
3. 能区分空载与带载下 MC 端口延迟  
4. 能输出 JSON / CSV / HTML  
5. 能通过阈值自动标记异常  
6. 能在报告中给出分段瓶颈定位结论

---

## 18. 总结

`mc-netprobe` 的核心不在于“再写一个测速工具”，而在于：

- 把总链路和分段链路统一到同一套执行流中
- 把空载与带载体验区分开
- 把网络指标与系统指标结合起来
- 把结果做成可重复、可比较、可归档的工程化输出

这套结构适合先由 Codex 生成 MVP，再逐步扩展。
